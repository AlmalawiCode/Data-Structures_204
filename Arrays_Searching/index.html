Part 1: Arrays - A Quick Reminder

You Should Already Know This!

Arrays are fundamental data structures that you learned in your programming courses. Let's quickly review what makes arrays special and understand their limitations.

What is an Array?

An array is a collection of elements stored in contiguous (consecutive) memory locations. Each element can be accessed directly using its index.

Example: Array of 10 Random Values

Key Properties of Arrays

  • Contiguous Memory: Elements are stored one after another in memory
  • Fixed Size: Once created, the size cannot change
  • Direct Access: Any element can be accessed instantly using its index
  • Same Type: All elements must be of the same data type

Part 2: What Does O(1) Mean?

You might be wondering...

We keep saying O(1) and O(n), but what do these actually mean?

Big O Notation - Simple Explanation

Big O notation describes how the time (or space) an algorithm needs grows as the input size increases.

  • O(1) - Constant Time: The operation takes the same amount of time regardless of how much data you have. It's the fastest possible!
  • O(n) - Linear Time: The time grows proportionally with the data size. Double the data = double the time.
  • O(log2n) - Logarithmic Time: The time grows very slowly even as data increases dramatically. Very efficient!

Visualizing O(1) - Constant Time

O(1) = "One Operation"

No matter if you have 10 items or 10 million items, O(1) takes the same time.

1,000 items

1,000,000 items

Why is Array Access O(1)?

Because of the address calculation formula:

Element Address = Base Address + (Index × Element Size)

The CPU can calculate this instantly, regardless of array size!

Part 3: Linear Search

What is Linear Search?

Linear Search is the simplest search algorithm. You start from the beginning and check each element one by one until you find what you're looking for.

When to Use Linear Search?

Linear search works on any array - sorted or unsorted. It's simple but can be slow for large datasets.

Step 4: Check index 3 → 33 ≠ 56 ❌

Step 5: Check index 4 → 91 ≠ 56 ❌

Worst Case Scenario

In algorithm analysis, we always consider the worst case. For linear search, the worst case is when the target element is at the very end of the array (or not present at all).

Linear Search Code

Linear Search - Java Implementation
/**
 * Linear Search Algorithm
 * Searches for a target value in an array by checking each element one by one.
 *
 * Time Complexity: O(n) - worst case checks all elements
 * Space Complexity: O(1) - no extra space needed
 *
 * @param arr    The array to search in
 * @param target The value we are looking for
 * @return       The index of target if found, -1 otherwise
 */
public static int linearSearch(int[] arr, int target) {

    // Loop through each element in the array
    for (int i = 0; i < arr.length; i++) {

        // Check if current element matches the target
        if (arr[i] == target) {
            return i;  // Found! Return the index
        }
    }

    // Target not found in the array
    return -1;
}

Part 4: Binary Search - The Fast Way!

Can We Do Better Than Linear Search?

Yes! But there's a requirement: the data must be sorted.

What is Binary Search?

Binary Search is a divide-and-conquer algorithm. Instead of checking every element, it:

  1. Looks at the middle element
  2. If the target is smaller, search the left half
  3. If the target is larger, search the right half
  4. Repeat until found (or not found)

Key Requirement

Binary Search ONLY works on SORTED data! If your data is not sorted, you must either:

  • Sort it first (which has its own cost), or
  • Use Linear Search instead

The Power of Binary Search

Each comparison eliminates HALF the data!

Binary Search Code

Binary Search - Java Implementation
/**
 * Binary Search Algorithm
 * Searches for a target value in a SORTED array using divide-and-conquer.
 *
 * Time Complexity: O(log₂n) - halves the search space each step
 * Space Complexity: O(1) - no extra space needed
 *
 * IMPORTANT: The array MUST be sorted for this to work!
 *
 * @param arr    The SORTED array to search in
 * @param target The value we are looking for
 * @return       The index of target if found, -1 otherwise
 */
public static int binarySearch(int[] arr, int target) {

    // Initialize low and high pointers
    int low = 0;
    int high = arr.length - 1;

    // Continue searching while the range is valid
    while (low <= high) {

        // Calculate the middle index
        int mid = (low + high) / 2;

        // Check if we found the target
        if (arr[mid] == target) {
            return mid;  // Found! Return the index
        }
        // If target is greater, ignore left half
        else if (arr[mid] < target) {
            low = mid + 1;  // Search in right half
        }
        // If target is smaller, ignore right half
        else {
            high = mid - 1;  // Search in left half
        }
    }

    // Target not found (low > high)
    return -1;
}

Remember These Rules:

  • If target > mid value → search right (set low = mid + 1)
  • If target < mid value → search left (set high = mid - 1)
  • If target = mid value → FOUND!
  • If low > high → NOT FOUND

Part 5: Understanding O(log2n)

Where Does log2n Come From?

Let's understand mathematically why binary search has time complexity O(log2n). The base is 2 because we divide by 2 (binary = two parts) at each step!

The Halving Process - Visualized

In binary search, we halve the search space with each comparison. Let's trace this step by step:

How Binary Search Reduces the Problem Size

Start
n items
We begin with n elements to search through
1
n / 2 items
After 1st comparison, we eliminate half. Only n/2 remains.
2
n / 4 = n / 22 items
After 2nd comparison, we halve again. Only n/4 remains.
3
n / 8 = n / 23 items
After 3rd comparison, only n/8 remains.
4
n / 16 = n / 24 items
After 4th comparison, only n/16 remains.
k
n / 2k items
General Pattern: After k comparisons, we have n / 2k items remaining!

When Do We Stop?

The search ends when only 1 item remains:

n / 2k = 1

Now we need to solve for k (the number of steps).

Mathematical Derivation: Solving for k

Step-by-Step Derivation of k = log2(n)
1
n / 2k = 1
Starting equation: After k steps, we have 1 item left.
2
2k · (n / 2k) = 1 · 2k
Multiply both sides by 2k: We multiply both the left and right sides by 2k to eliminate the denominator.
Algebraic Rule: If we multiply both sides of an equation by the same value, the equation remains balanced.
3
n = 2k
Simplify: On the left, 2k cancels with 2k in the denominator, leaving n. On the right, 1 · 2k = 2k.
Cancellation Rule: (a/b) × b = a (the b's cancel out)
4
log2(n) = log2(2k)
Take log base 2 of both sides: We use log base 2 because we are dividing by 2 (binary).
Logarithm Definition: If bx = y, then logb(y) = x
In other words: "log base b of y asks: what power do we raise b to get y?"
5
log2(n) = k × log2(2)
Apply the Power Rule of Logarithms: The exponent k comes down as a multiplier.
Power Rule: logb(xk) = k × logb(x)
This rule says: "the log of a power equals the exponent times the log of the base"
6
log2(n) = k × 1
Simplify log2(2): Log base 2 of 2 equals 1.
Identity Rule: logb(b) = 1
Why? Because b1 = b. So "what power of b gives b?" The answer is 1.
7
k = log2(n)
Final Result: The number of steps k equals log base 2 of n!
KEY POINT

The number of steps k required to reduce n items to 1 by repeatedly halving is:

k = log2(n)

Therefore, Binary Search runs in O(log2n) time.

Note: In Big O notation, we often write O(log n) without the base because changing the base only changes the constant factor, not the growth rate. However, for binary search, the base is specifically 2 because we divide by 2!

Why Base 2? Because It's BINARY!

The Connection to "Binary"

The word "binary" means two. In binary search:

  • We divide the array into 2 parts
  • We keep 1 part and discard 1 part
  • We divide by 2 at each step

This is why the logarithm has base 2 - because we're doing binary (two-way) division!

Practical Examples: How Many Steps?

Number of Items (n) Linear Search (n steps) Binary Search (log2n steps) Calculation
8 8 3 log2(8) = log2(23) = 3
16 16 4 log2(16) = log2(24) = 4
1,024 1,024 10 log2(1024) = log2(210) = 10
1,000,000 1,000,000 ~20 log2(1,000,000) ≈ 19.93
1,000,000,000 1,000,000,000 ~30 log2(1,000,000,000) ≈ 29.90

Amazing Efficiency!

To search through 1 billion items:

  • Linear Search: Up to 1,000,000,000 comparisons!
  • Binary Search: Only about 30 comparisons!

That's the power of O(log2n)!

The Trade-Off

Binary search is much faster, BUT:

  • Data must be sorted first
  • Sorting takes time (usually O(n log n))
  • Worth it if you search many times on the same data
