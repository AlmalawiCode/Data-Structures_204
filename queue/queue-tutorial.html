1. What is a Queue?

Definition: A Queue is a linear data structure that stores elements in a sequential manner where insertion happens at one end called the REAR and deletion happens at the other end called the FRONT. It follows the FIFO (First In, First Out) principle — the first element added is the first one removed.

Queue in the Family of Data Structures

Data structures are divided into two broad categories:

  • Physical Data Structures — structures that actually exist in memory and have a direct hardware representation. Examples: Arrays and Linked Lists.
  • Logical Data Structures — structures that define a logical behavior for organizing and accessing data. They cannot be built without a physical data structure underneath them. Examples: Stack, Queue, Tree, and Graph.

Queue is a Logical Data Structure — it describes a concept (FIFO access policy), but to actually store data it must be implemented on top of an Array or a Linked List.

Real-World Examples of Queues

👥 Supermarket Checkout

Customers line up at the checkout. The first customer to join the line is the first to be served.

✈️ Airport Check-In

Passengers queue at the check-in counter. First to arrive, first to check in.

📈 Print Queue

Documents sent to a printer are printed in the order they were submitted — first submitted, first printed.

📱 Call Center

Incoming calls are placed in a queue. The first caller is connected to an agent first.

Key Characteristics of a Queue

  • Linear Data Structure: Elements are arranged in a sequential order
  • Two-End Access: Addition at the REAR, removal at the FRONT
  • FIFO Policy: First In, First Out — the element that waited the longest is served first
  • Two Pointers: front tracks the first element, rear tracks the last element

Queue vs. Stack — The Key Difference

Both Stack and Queue are linear, logical data structures. The critical difference is their access policy:

  • Stack → LIFO (Last In, First Out) — one end (TOP)
  • Queue → FIFO (First In, First Out) — two ends (FRONT and REAR)

Think of it this way: a stack is like a pile of plates (you take from the top), while a queue is like a line of people (the first person in line leaves first).

2. Queue as an Abstract Data Type (ADT)

Why is Queue an Abstract Data Type?

A Queue is an Abstract Data Type (ADT) because it defines the behavior through a set of operations without specifying how those operations are implemented internally. The user of the queue sees only the interface: enqueue here, dequeue from there — they do not need to know whether an array or a linked list is used underneath.

Logical vs. Physical — Summary

  • Queue cannot exist alone in memory — it is a logical concept.
  • It must be implemented on top of a physical structure (Array or Linked List).
  • The behavior (FIFO) stays the same no matter which physical structure you choose.
  • This separation between behavior and implementation is what makes Queue an ADT.

Access Policy: FIFO

FIFO = First In, First Out

The element that was inserted first is the one that gets removed first. This is the fundamental rule (access policy) of every queue, regardless of how it is implemented.

3. Queue Operations

The six fundamental operations of a queue are:

3.1 enqueue(element) — Add to Rear

Purpose: Adds an element to the REAR of the queue

Analogous to: push() in a Stack

Time Complexity: O(1) for unrealistic and circular; O(n) for shifting (realistic)

Algorithm — enqueue(element)

  1. Check if the queue is full. If full → report overflow error and stop.
  2. Advance the rear pointer: rear = rear + 1 (or rear = (rear+1) % SIZE for circular)
  3. Store the element at the new rear position: queue[rear] = element
  4. If this is the first element being added (front was −1), also set front = 0

3.2 dequeue() — Remove from Front

Purpose: Removes and returns the element at the FRONT of the queue

Analogous to: pop() in a Stack

Time Complexity: O(1) for unrealistic and circular; O(n) for shifting

Algorithm — dequeue()

  1. Check if the queue is empty. If empty → report underflow error and stop.
  2. Save the element at the front: temp = queue[front]
  3. Advance the front pointer: front = front + 1 (or front = (front+1) % SIZE for circular)
  4. If front > rear, the queue is now empty — reset both pointers to −1
  5. Return temp

3.3 peek() — Inspect Front

Purpose: Returns the element at the front without removing it

Analogous to: top() in a Stack

Time Complexity: O(1)

3.4 isEmpty()

Purpose: Returns true if the queue has no elements

Condition: front == -1 (or front > rear in the linear case)

3.5 isFull()

Purpose: Returns true if the queue cannot hold more elements

Condition: rear == SIZE - 1 for unrealistic/realistic; (rear + 1) % SIZE == front for circular

3.6 clear()

Purpose: Empties the queue by resetting both pointers

Action: front = rear = -1

enqueue(e) Add e at rear queue[(rear+1)%N] = e tail.next = newNode

4. Array-Based Implementation

Since a Queue is a logical data structure, it must be implemented using a physical structure. Let us start with an Array. We use an array of fixed size (e.g., 5 elements) and maintain two integer pointers:

  • front — index of the first (oldest) element; starts at −1
  • rear — index of the last (newest) element; starts at −1

When both are −1, the queue is empty.

There are three strategies for array-based queue implementation. Each has different trade-offs:

  1. Unrealistic — simple but wastes space
  2. Realistic (Shifting) — uses all space but slow
  3. Circular — best of both worlds using modulo arithmetic ✓

4.1 Strategy 1 — Unrealistic (Simple Linear)

In this strategy, we simply advance both pointers as elements are added and removed. We never shift elements back. The pointer only moves forward — it never goes back to the start.

How it works

  • Initially: front = rear = −1
  • enqueue: increment rear, place element at queue[rear]
  • dequeue: save queue[front], increment front
  • isFull: rear == SIZE − 1

The Problem — Why It Is Called "Unrealistic"

Imagine a queue of size 5. You enqueue 5 elements (indices 0–4). Now the queue is full (rear = 4). You dequeue 3 elements (front = 3). Logically, 3 slots are now free (indices 0, 1, 2). But the condition rear == SIZE − 1 is still true, so the queue reports FULL even though 3 positions are available!

This wasted space can never be reclaimed. The queue will refuse new insertions even though it has free spots. This is why this strategy is called unrealistic — it is not practical for real use.

4.2 Strategy 2 — Realistic (Shifting)

A student observes the problem above and suggests: "Why don't we shift all elements toward index 0 after each dequeue?" This is the shifting strategy. After removing the front element, every remaining element shifts one position to the left, and front always stays at 0.

How Shifting Works

  • front is always 0
  • dequeue: remove queue[0], shift all elements left (queue[i] = queue[i+1]), decrement rear
  • No space is ever wasted — all free positions are always at the right end

The Trade-Off — Why Shifting Hurts Performance

Every dequeue operation requires shifting all remaining elements one position to the left. If there are n elements in the queue, that is O(n) work per dequeue — not O(1). This can be very slow for large queues. The enqueue is still O(1), but dequeue is O(n).

4.3 Strategy 3 — Circular Queue (Best Solution)

Circular Queue: Instead of moving forward in a straight line, the array is treated as a circle. When rear reaches the last index, it wraps around to index 0 using the modulo operator (%). This eliminates wasted space and keeps all operations at O(1).

The Key Formula — Modulo Arithmetic

  • Enqueue: rear = (rear + 1) % SIZE
  • Dequeue: front = (front + 1) % SIZE
  • isEmpty: front == -1 (or use a count variable)
  • isFull: (rear + 1) % SIZE == front

The modulo operator makes the index wrap around. If SIZE = 5 and rear = 4, then (4 + 1) % 5 = 0 — rear jumps back to the start!

Important Cases in Circular Queue

Situation Condition Meaning
Empty queue front == -1 No elements; both pointers reset to −1
Full queue (rear + 1) % SIZE == front One slot is always kept empty as a sentinel to distinguish full from empty
One element front == rear front and rear point to the same slot
Wrap-around rear = (rear+1) % SIZE rear wraps from last index back to 0

Comparison of the Three Strategies

❌ Unrealistic

enqueue: O(1)

dequeue: O(1)

Problem: Space is wasted — positions freed by dequeue can never be reused.

⚠️ Realistic (Shifting)

enqueue: O(1)

dequeue: O(n) — shift all elements!

Problem: No wasted space, but slow dequeue for large queues.

✓ Circular (Best)

enqueue: O(1)

dequeue: O(1)

Advantage: Wraps around with modulo — no wasted space, no shifting needed!

5. Java Implementation

5.1 Circular Queue Using Array

▶ CircularQueue.java — Array-based Circular Queue ▼
public class CircularQueue { private int[] queue; private int front, rear, count, size; // Constructor public CircularQueue(int size) { this.size = size; queue = new int[size]; front = rear = -1; count = 0; } // isEmpty — O(1) public boolean isEmpty() { return count == 0; } // isFull — O(1) public boolean isFull() { return count == size; } // enqueue — Add to REAR — O(1) public void enqueue(int value) { if (isFull()) { System.out.println("Queue Overflow! Cannot enqueue " + value); return; } if (isEmpty()) front = 0; // first element rear = (rear + 1) % size; // wrap rear queue[rear] = value; count++; } // dequeue — Remove from FRONT — O(1) public int dequeue() { if (isEmpty()) { System.out.println("Queue Underflow! Cannot dequeue"); return -1; } int removed = queue[front]; count--; if (isEmpty()) { // queue is now empty front = rear = -1; } else { front = (front + 1) % size; // wrap front } return removed; } // peek — View front element — O(1) public int peek() { if (isEmpty()) { System.out.println("Queue is empty!"); return -1; } return queue[front]; } // clear — Reset queue public void clear() { front = rear = -1; count = 0; } // display contents public void display() { if (isEmpty()) { System.out.println("Queue is empty"); return; } System.out.print("Queue [FRONT→REAR]: "); int i = front; for (int k = 0; k < count; k++) { System.out.print(queue[i] + " "); i = (i + 1) % size; } System.out.println(); } }

5.2 Queue Using Linked List

▶ LinkedQueue.java — Linked List-based Queue ▼
// Node class class Node { int data; Node next; Node(int data) { this.data = data; next = null; } } public class LinkedQueue { private Node front, rear; private int count; public LinkedQueue() { front = rear = null; count = 0; } public boolean isEmpty() { return front == null; } // enqueue — O(1): always insert at tail public void enqueue(int value) { Node newNode = new Node(value); if (isEmpty()) { front = rear = newNode; } else { rear.next = newNode; rear = newNode; } count++; } // dequeue — O(1): always remove from head public int dequeue() { if (isEmpty()) { System.out.println("Queue Underflow!"); return -1; } int removed = front.data; front = front.next; if (front == null) rear = null; // queue is empty count--; return removed; } public int peek() { if (isEmpty()) return -1; return front.data; } }

Why Linked List Implementation is Easy

With a linked list, there is no concept of "full" — the list grows dynamically as needed. There is no wasted space, no wrap-around logic, and no shifting. The trade-off is additional memory overhead per node (the next pointer). For most practical purposes, the circular array or linked list implementations are both excellent choices.
