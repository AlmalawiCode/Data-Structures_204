3 عودة إلى أحمد وعمر — لماذا المقارنة غير عادلة؟

ثلاثة أسباب تجعل قياس الوقت وحده غير موثوق

فكّر الآن في أحمد وعمر. ماذا لو كان أحد هذه السيناريوهات صحيحًا؟

💻

أجهزة (حواسيب) مختلفة

ربما كان أحمد يُشغّل خوارزميته على محطة عمل عالية الأداء وحديثة، بينما يستخدم عمر حاسوبًا محمولًا قديمًا وبطيئًا. حاسوب أحمد هو "الرصيف الصلب" — بالطبع سيكون أسرع!

👨‍💻

مبرمجون مختلفون

ربما كانت خوارزمية عمر أكثر كفاءةً في الواقع، لكن تنفيذه بلغة Java كان ضعيفًا. أحمد مبرمج أكثر خبرةً وكتب كودًا أنظف وأكثر تحسينًا، لذا نسخته تعمل أسرع.

🔨

أدوات مختلفة

حتى على نفس الحاسوب، يمكن لغات البرمجة المختلفة أو المُصرِّفات أو بيئات التشغيل أن تُسبّب فروقًا زمنية كبيرة — حتى لنفس منطق الخوارزمية تمامًا.

💡 المشكلة الحقيقية
كيف نحكم بعدالة على أي خوارزمية أفضل؟
نحتاج إلى قياس مستقل عن كل هذه العوامل الخارجية

🔎 فكرة الحل: نجعل نفس المبرمج المحترف ينفّذ كلتا الخوارزميتين ويُشغّلهما على نفس الحاسوب. سيكون هذا أعدل — لكنه لا يزال ليس أفضل نهج. نحتاج إلى شيء أكثر جوهريةً وعدالةً شاملة.

4 ما نحتاجه — الحَكَم العادل

قياس شامل وموضوعي بحق

لقياس الخوارزميات ومقارنتها بعدالة، نحتاج إلى منهجية مستقلة تمامًا عن كل العوامل الخارجية:

💻
مستقلة عن الحاسوب سواء شغّلتها على حاسوب قديم أو حاسوب خارق، يجب أن يبقى الحكم واحدًا
👨‍💻
مستقلة عن المبرمج مهارة الشخص أو خبرته في كتابة الكود يجب ألا تؤثر في النتيجة
🔨
مستقلة عن لغة البرمجة Java أو Python أو C++ أو أي لغة أخرى لا ينبغي أن تغيّر القياس الجوهري
📊
مبنية على سلوك الخوارزمية مع نمو المدخلات ماذا يحدث عندما يتضاعف n؟ عندما يكون n مليونًا مقابل مليار؟ هذا هو المقياس الحقيقي للخوارزمية
🌟

الإجابة: تحليل الخوارزميات ورمز O الكبير (Big O)

ندرس عدد العمليات التي تؤديها الخوارزمية كدالة في حجم المدخلات n. يمنحنا هذا مقياسًا رياضيًا مستقلًا عن الحاسوب والمبرمج، يعكس الكفاءة الحقيقية للخوارزمية.

5 اللبنة الأساسية — ما هو O(1)؟

أصغر وحدة عمل في أي خوارزمية

لعدّ عدد العمليات التي تؤديها الخوارزمية، يجب أولًا أن نتفق على ما يُعدّ عملية واحدة. في علوم الحاسوب، نعرّف أي جملة برمجية بسيطة مفردة على أنها تستغرق وحدة زمنية واحدة — نسمي هذا O(1)، أو الزمن الثابت.

💡 اتفاق مهم: لا يمكننا قياس الزمن الدقيق للجملة البرمجية (يتغير مع كل حاسوب). لذلك نتفق: كل جملة مفردة، بصرف النظر عن الحاسوب الذي يُشغّلها، تُحسب على أنها 1 عملية = O(1). هذه وحدتنا العادلة والشاملة.

أمثلة على الجمل المفردة — كل منها O(1):

i = i + 1;
←
O(1)
1 عملية
حتى لو ترجمت المعالجة i = i + 1 داخليًا إلى عدة تعليمات دقيقة، نحن نتفق على عدّها عملية واحدة. هذا يجعل تحليلنا مستقلًا عن الأجهزة.
x = arr[5];
←
O(1)
1 عملية
الوصول إلى عنصر المصفوفة بالمؤشر يستغرق دائمًا عملية واحدة، سواء كانت المصفوفة تحتوي على 10 عناصر أو 10 ملايين. يستلزم نفس الجهد الثابت.
sum = a + b;
←
O(1)
1 عملية
العمليات الحسابية وتعيينات المتغيرات والمقارنات — جميعها O(1). النتيجة لا تعتمد على حجم أي مدخل n.

✅ الفكرة المحورية: O(1) لا يعني "يستغرق 1 ثانية" أو "1 مللي ثانية". يعني "يستغرق مقدارًا ثابتًا من الوقت — نفس المقدار بغض النظر عن حجم المسألة". هذه وحدة قياسنا الشاملة والعادلة.

6 من O(1) إلى O(n) — كيف تعمل الحلقات

البناء التدريجي من العمليات المفردة إلى الخوارزميات

الآن بعد أن عرفنا أن O(1) هي لبنتنا الأساسية، دعنا نرى ما يحدث عندما نضع تلك العملية داخل حلقة تكرار. هكذا نبدأ بعدّ إجمالي العمل الذي تؤديه الخوارزمية.

int x = 0; x = x + 1; // ← جملة واحدة
إجمالي العمليات:
O(1)
= O(1)
for (int i = 0; i < 10; i++) { x = x + 1; // تُنفَّذ 10 مرات }
عملية O(1) واحدة تتكرر 10 مرات:
O(1) O(1) O(1) O(1) O(1) O(1) O(1) O(1) O(1) O(1)
O(1) × 10 = O(10)
for (int i = 0; i < n; i++) { x = x + 1; // تُنفَّذ n مرة }
عملية O(1) واحدة تتكرر n مرة:
O(1) O(1) O(1) … O(1)
O(1) × n = O(n)

💡 النمط: حلقة تكرار تعمل n مرة، مع عمل O(1) بداخلها، تعطينا O(1) + O(1) + … + O(1) (n مرة) = O(n). يزيد عدد العمليات بشكل خطي مع حجم المدخلات. ضاعف المدخلات ← ضاعف العمل.

ماذا عن الحلقات المتداخلة؟

for (int i = 0; i < n; i++) { // n مرة for (int j = 0; j < n; j++) { // n مرة x = x + 1; // O(1) } }
الحلقة الداخلية تعمل n مرة لكل تكرار من الحلقة الخارجية البالغة n تكرارًا:
n × n
= n² عملية = O(n²)

✅ الفكرة الكبرى: لا نحتاج إلى معرفة زمن التشغيل الدقيق بالثواني. نعدّ عدد العمليات كدالة رياضية في حجم المدخلات n. هذا العدد ثابت بصرف النظر عن الحاسوب أو المبرمج أو اللغة. هذا هو حَكَمنا العادل.

7 سلوكيات الخوارزميات — فئات التعقيد التسع

كيف تنمو الخوارزميات المختلفة مع زيادة حجم المدخلات

لا نحتاج إلى معرفة العدد الدقيق للعمليات. ما يهمنا هو نمط السلوك — كيف ينمو عدد العمليات مع ازدياد حجم المدخلات n. نسمي هذا رمز O الكبير (Big O). كل خوارزمية تقع ضمن إحدى هذه الفئات التسع. استكشف كل فئة أدناه لترى رسمًا بيانيًا حقيقيًا وجدولًا يوضح بالضبط مدى سرعة نمو كل تعقيد.

O(1) — ثابت O(log n) — لوغاريتمي O(√n) — جذر تربيعي O(n) — خطي O(n·log n) — خطي-لوغاريتمي O(n²) — تربيعي O(n³) — تكعيبي O(2ⁿ) — أسي O(n!) — مضروب O(nⁿ) — n أس n
O(1) الزمن الثابت (Constant Time)

ما يعنيه: تأخذ الخوارزمية دائمًا نفس عدد العمليات، بصرف النظر عن حجم المدخلات. حجم المسألة ببساطة لا يؤثر في زمن التشغيل.

مثال حقيقي: الوصول إلى عنصر بالمؤشر — arr[5] سريع بنفس الدرجة سواء كانت المصفوفة تحتوي على 10 أو 10 ملايين عنصر.

لاحظ في الرسم البياني: خط أفقي مستقيم تمامًا. مع نمو n من 1 إلى 100، تبقى العمليات عند 1 بالضبط.

Java — O(1) الزمن الثابت ▼
// O(1) — Constant Time: operations do NOT depend on n int getFirst(int[] arr) { return arr[0]; // Always exactly 1 operation }

f(n) = 1  —  العمليات ثابتة بغض النظر عن حجم المدخلات

حجم المدخلات (n) العمليات f(n) = 1
O(log n) الزمن اللوغاريتمي (Logarithmic Time)

ما يعنيه: في كل مرة يتضاعف n، يزيد عدد العمليات بمقدار 1 فقط. ينمو، لكن ببطء شديد.

مثال حقيقي: البحث الثنائي — البحث في مصفوفة مرتبة من مليون عنصر لا يحتاج سوى نحو 20 مقارنة!

لاحظ في الرسم البياني: المنحنى يرتفع بحدة في البداية ثم ينحني ويكاد يصبح مستقيمًا. حتى لـ 100 مليون عنصر، log₂(n) لا يتجاوز نحو 27 عملية.

Java — O(log n) الزمن اللوغاريتمي ▼
// O(log n) — Logarithmic: n is halved each iteration void logExample(int n) { while (n > 1) { n = n / 2; // n is cut in half → only log₂(n) iterations needed } }

f(n) = log₂(n)  —  الرسم: n = 1 إلى 1,000  |  الجدول: حتى 100 مليون

حجم المدخلات (n) العمليات f(n) = log₂(n)
O(√n) زمن الجذر التربيعي (Square Root Time)

ما يعنيه: تنمو العمليات بمقدار الجذر التربيعي لحجم المدخلات — أبطأ بكثير من الخطي. عندما يتضاعف n أربع مرات، تتضاعف العمليات مرتين فقط.

مثال حقيقي: اختبار الأولية — للتحقق من أن n أولي تحتاج فقط اختبار المقسومات حتى √n، وليس حتى n.

لاحظ في الرسم البياني: المنحنى يرتفع بسرعة في البداية ثم يتسطح تدريجيًا. عند n = 10,000، نحتاج 100 عملية فقط بدلًا من 10,000.

Java — O(√n) زمن الجذر التربيعي ▼
// O(√n) — Square Root: loop runs only up to √n boolean isPrime(int n) { if (n < 2) return false; for (int i = 2; i * i <= n; i++) { // loop runs √n times if (n % i == 0) return false; } return true; } // Condition i*i ≤ n stops the loop at exactly √n iterations

f(n) = √n  —  ينمو أبطأ بكثير من الخطي

حجم المدخلات (n) العمليات f(n) = √n
O(n) الزمن الخطي (Linear Time)

ما يعنيه: تنمو العمليات بشكل متناسب مع حجم المدخلات. إذا تضاعف n، تضاعفت العمليات أيضًا. زيادة منتظمة ومستمرة.

مثال حقيقي: البحث الخطي — مسح قائمة غير مرتبة عنصرًا بعنصر من البداية إلى النهاية.

لاحظ في الرسم البياني: خط قطري مستقيم تمامًا. كل زيادة في n تضيف نفس العدد الثابت من العمليات.

Java — O(n) الزمن الخطي ▼
// O(n) — Linear: one pass through all n elements int linearSearch(int[] arr, int target) { for (int i = 0; i < arr.length; i++) { // runs n times if (arr[i] == target) return i; } return -1; }

f(n) = n  —  العمليات تنمو في خط مستقيم تمامًا

حجم المدخلات (n) العمليات f(n) = n
O(n·log n) الزمن الخطي-اللوغاريتمي (Linearithmic Time)

ما يعنيه: تنمو العمليات بشكل أسرع قليلًا من الخطي لكن أبطأ بكثير من التربيعي. عندما يتضاعف n، تزيد العمليات بأكثر من الضعف بقليل.

مثال حقيقي: الترتيب بالدمج (Merge Sort)، ترتيب الكومة (Heap Sort) — تقسيم على O(log n) مستوى مع عمل O(n) في كل مستوى، فيكون الإجمالي O(n·log n).

لاحظ في الرسم البياني: يبدو شبه خطي عند القيم الصغيرة لـ n، ثم ينحني بلطف إلى الأعلى. هذه أفضل سرعة ممكنة لخوارزميات الترتيب القائمة على المقارنة.

Java — O(n·log n) الزمن الخطي-اللوغاريتمي ▼
// O(n·log n) — the loop pattern seen in merge sort // outer loop: n iterations; inner loop halves j → log₂(n) iterations void nLogNExample(int n) { for (int i = 1; i <= n; i++) { // runs n times for (int j = n; j >= 1; j = j / 2) { // halves → log₂(n) iterations process(i, j); } } } // Total: n × log₂(n) operations

f(n) = n · log₂(n)  —  أعلى بقليل من الخطي، "النقطة الذهبية" للترتيب

حجم المدخلات (n) العمليات f(n) = n · log₂(n)
O(n²) الزمن التربيعي (Quadratic Time)

ما يعنيه: تنمو العمليات بمربع حجم المدخلات. إذا تضاعف n، تتضاعف العمليات 4 أضعاف!

مثال حقيقي: ترتيب الفقاعة (Bubble Sort)، ترتيب الانتقاء (Selection Sort) — حلقتان متداخلتان تعمل كل منهما n مرة، فيكون الإجمالي n × n = n² عملية.

لاحظ في الرسم البياني: المنحنى ينحني بحدة إلى الأعلى (قطع مكافئ). عند n=100، نحصل على 10,000 عملية — قارن ذلك بـ 100 فقط لـ O(n).

Java — O(n²) الزمن التربيعي ▼
// O(n²) — Quadratic: two nested loops void printPairs(int[] arr, int n) { for (int i = 0; i < n; i++) { // outer loop: n times for (int j = 0; j < n; j++) { // inner loop: n times process(arr[i], arr[j]); } } } // Total: n × n = n² operations

f(n) = n²  —  قطع مكافئ: ينمو أسرع بكثير من الخطي

حجم المدخلات (n) العمليات f(n) = n²
O(n³) الزمن التكعيبي (Cubic Time)

ما يعنيه: تنمو العمليات بمكعب حجم المدخلات. أشد انفجارًا من التربيعي. إذا تضاعف n، تتضاعف العمليات 8 أضعاف!

مثال حقيقي: ضرب المصفوفات بالطريقة الساذجة — ثلاث حلقات متداخلة تعمل كل منها n مرة.

لاحظ في الرسم البياني: المنحنى يرتفع بصورة أحدّ من O(n²). عند n=20 نحصل على 8,000 عملية. عند n=50 نحصل على 125,000 — لاحظ مدى سرعة الانفجار.

Java — O(n³) الزمن التكعيبي ▼
// O(n³) — Cubic: three nested loops void tripleLoop(int n) { for (int i = 0; i < n; i++) { // n times for (int j = 0; j < n; j++) { // n times for (int k = 0; k < n; k++) { // n times process(i, j, k); } } } } // Total: n × n × n = n³ operations

f(n) = n³  —  أحدّ من التربيعي، ينمو بسرعة كبيرة

حجم المدخلات (n) العمليات f(n) = n³
O(2ⁿ) الزمن الأسي (Exponential Time)

ما يعنيه: تتضاعف العمليات مع كل زيادة مفردة في n. هذا أشد نمو انفجاري ممكن. كل +1 في n يضرب العمليات في 2.

مثال حقيقي: فيبوناتشي التعاودي بدون حفظ (Memoization)، توليد جميع المجموعات الجزئية، تكسير كلمات المرور بالقوة الخام.

لاحظ في الرسم البياني: يبدو المنحنى شبه مستقيم، ثم يرتفع فجأة كالجدار. عند n=30 لدينا أكثر من مليار عملية — لهذا الخوارزميات الأسية غير عملية لأي حجم مدخلات معقول.

Java — O(2ⁿ) الزمن الأسي ▼
// O(2ⁿ) — Exponential: two recursive calls each time int fibonacci(int n) { if (n <= 1) return n; // base case return fibonacci(n-1) + fibonacci(n-2); // 2 calls → branches double every level → 2ⁿ }

f(n) = 2ⁿ  —  العمليات تتضاعف مع كل +1 في n

حجم المدخلات (n) العمليات f(n) = 2ⁿ
O(n!) زمن المضروب (Factorial Time)

ما يعنيه: تنمو العمليات بمقدار مضروب n: n × (n−1) × … × 1. النمو كارثي السرعة — عند n=20 لدينا 2.4 كوادريليون عملية.

مثال حقيقي: التبديل بالقوة الخام — توليد كل ترتيب ممكن لـ n عنصر (مثل مسألة البائع المتجول حلًا ساذجًا).

لاحظ في الرسم البياني: حتى لأصغر قيم n (حتى 10)، تنفجر القيمة بشكل درامي. خوارزميات المضروب عملية فقط لمدخلات صغيرة جدًا.

Java — O(n!) زمن المضروب ▼
// O(n!) — Factorial: generating all permutations void permute(int[] arr, int start) { if (start == arr.length) { process(arr); // one complete permutation return; } for (int i = start; i < arr.length; i++) { swap(arr, start, i); // try each element permute(arr, start + 1); // recurse on remaining swap(arr, start, i); // backtrack } } // Total recursive calls: n! — one per permutation

f(n) = n!  —  نمو كارثي، عملي فقط لـ n صغير جدًا

حجم المدخلات (n) العمليات f(n) = n!
O(nn) زمن n أس n (n-to-the-n Time)

ما يعنيه: فئة التعقيد الأسوأ نظريًا. العمليات تساوي n مضروبًا بنفسه n مرة. ينمو أسرع من المضروب — n! < nn لكل n > 1.

مثال حقيقي: الاستعداد الشامل حيث كل موقع من n موقع يمكنه بشكل مستقل أن يأخذ أي من n قيمة. في الغالب حد نظري علوي، نادرًا ما يُرى عمليًا.

لاحظ في الرسم البياني: حتى عند n=8، تتجاوز القيمة 16 مليونًا. هذه الفئة تُعرّف السقف الرياضي لتحليل التعقيد.

Java — O(nn) المفهوم ▼
// O(nⁿ) — each of n positions can independently take any of n values // Example: enumerate all n-digit strings over an alphabet of size n void enumerateAll(int n, int[] curr, int pos) { if (pos == n) { process(curr); // one complete assignment return; } for (int v = 0; v < n; v++) { // n choices at each position curr[pos] = v; enumerateAll(n, curr, pos + 1); } } // Total: n × n × … × n (n times) = nⁿ

f(n) = nn  —  الحد الأسوأ نظريًا، ينمو أسرع من المضروب

حجم المدخلات (n) العمليات f(n) = nn

👀 لنعمل على مثال كامل وملموس من البداية إلى النهاية. أدناه دالة Java تُسمى counter(n). تستخدم ثلاثة أقسام حلقات منفصلة، يؤدي كل منها عددًا مختلفًا من العمليات وفقًا لـ n. قبل تحليلها، دعنا أولًا نُشغّلها ونرى الناتج.

Java — PolynomialCounter.java ▼
public class PolynomialCounter { public static int counter(int n) { int c = 0; // Section A — 5n² for (int a = 1; a <= 5; a++) { for (int i = 1; i <= n; i++) { for (int j = 1; j <= n; j++) { c++; } } } // Section B — 5n for (int a = 1; a <= 5; a++) { for (int i = 1; i <= n; i++) { c++; } } // Section C — 5 (constant) for (int i = 1; i <= 5; i++) { c++; } return c; } public static void main(String[] args) { System.out.println(counter(5)); // What does this print? } }

👀 عندما يكون n = 5، ماذا تطبع counter(5)؟

counter(5) → 155 عملية تم إحصاؤها
💡 الفكرة المحورية
هل يمكننا التنبؤ بـ 155 دون تشغيل الكود؟
نعم — بالتعبير عن الخوارزمية كـ دالة كثيرات الحدود في n

يُسهم كل قسم من أقسام الدالة counter(n) بحدٍّ واحد في معادلة كثيرات الحدود:

القسم A — 3 حلقات متداخلة
الخارجية: a = 1 إلى 5 (5 مرات ثابتة)
الداخليتان: i = 1 إلى n، j = 1 إلى n
← 5 × n × n = 5n² زيادة
5n²
القسم B — 2 حلقتان متداخلتان
الخارجية: a = 1 إلى 5 (5 مرات ثابتة)
الداخلية: i = 1 إلى n
← 5 × n = 5n زيادة
5n
القسم C — 1 حلقة (ثابتة)
i = 1 إلى 5 (لا تعتمد أبدًا على n)
دائمًا 5 زيادات بالضبط تمامًا
5
إجمالي العمليات = مجموع الأقسام الثلاثة في الدالة counter(n)
f(n) = 5n² + 5n + 5
✅ إثبات: عوّض n = 5 في كثيرات الحدود
f(5) = 5(5²) + 5(5) + 5 = 5(25) + 25 + 5 = 125 + 25 + 5 = 155

استنتجنا للتو أن counter(n) = O(n²) بالحدس — احتفظ بالحد الأكبر واحذف الباقي. هذا اختصار مفيد، لكنه ليس إثباتًا.

لجعله صارمًا، يجب إيجاد قيم محددة لـ C وn₀ وإثبات أن المتراجحة صحيحة. هذا بالضبط ما نفعله في الخطوة التالية.

💡

ماذا يعني "O(n²)" بصريًا؟

عندما نقول f(n) هو O(g(n))، نعني أنه يمكننا إيجاد مضاعف من g(n) يكبر في النهاية أسرع من f(n) ويبقى فوقه إلى الأبد بعد نقطة معينة n₀.

في حالتنا:

  • المنحنى الأحمر = دالتنا f(n) = 5n² + 5n + 5
  • المنحنى الأخضر = الحد العلوي C · n² — نحتاج إيجاد C يبقى فوق المنحنى الأحمر
  • n₀ = النقطة التي يكون بعدها المنحنى الأخضر دائمًا ≥ المنحنى الأحمر

قد يتقاطع المنحنيان عند n₀ — ما يهم هو أنه لكل n ≥ n₀، يهيمن المنحنى الأخضر.

📖

التعريف الرسمي لرمز O الكبير (Big O)

التعريف الرسمي
f(n) is O(g(n))   ⟺   ∃ positive constants C and n₀ such that:
f(n) ≤ C · g(n)    for all n ≥ n₀
C
ثابت موجب مضاعف يجب إيجاده
n₀
نقطة البداية التي يسري منها الحد
f(n) ≤ C·g(n)
المتراجحة التي يجب إشباعها

هدفنا: إثبات أن 5n² + 5n + 5 ≤ C · n² لبعض C الموجب ولكل n ≥ n₀. نحتاج إيجاد القيم الدقيقة لـ C وn₀.

1

الخطوة 1 — إيجاد الثابت C

الاستراتيجية (الطريقة القياسية): استبدل كل حد في f(n) بشيء أكبر يستخدم n² فقط. يمنحنا هذا حدًّا عليًا نظيفًا من الشكل C · n².

f(n)  =  5n²  +  5n  +  5 الدالة الأصلية
≤ 5n²  +  5n²  +  5n² 5n ≤ 5n² و 5 ≤ 5n²  لكل n ≥ 1
= (5 + 5 + 5) · n²  =  15n² جمع المعاملات
لماذا 5n ≤ 5n²؟ لأنه لكل n ≥ 1، n ≤ n²، إذن 5n ≤ 5n².  ✓
لماذا 5 ≤ 5n²؟ لأنه لكل n ≥ 1، 1 ≤ n²، إذن 5 ≤ 5n².  ✓
🎯
وجدنا C
C = 15   →   5n² + 5n + 5 ≤ 15n²
2

الخطوة 2 — إيجاد n₀ (التحقق من المتراجحة)

الاستراتيجية: نبدأ دائمًا بتجربة n = 1. إذا كانت المتراجحة f(1) ≤ C · g(1) صحيحة، فإن n₀ = 1. وإلا، نجرب n = 2, 3, … حتى تتحقق.

نجرب n = 1:
الطرف الأيسر: 5(1²) + 5(1) + 5  =  5 + 5 + 5  =  15
الطرف الأيمن: 15 · (1²)  =  15
⇒ 15 ≤ 15 ✓ صحيح (المساواة قائمة!)
🔎 التحقق عند n = 1
f(1) = 5·1² + 5·1 + 5 = 5 + 5 + 5 = 15
C·g(1) = 15·1² = 15

15 ≤ 15 → صحيح ✓

المتراجحة صحيحة عند n = 1، إذن n₀ = 1.

📍
وجدنا n₀
n₀ = 1   →   الحد ساري لكل n ≥ 1

💡 ملاحظة: عند n = 1 المنحنيان متساويان (كلاهما = 15). لكل n > 1، يكبر الحد C·n² = 15n² بصورة أسرع من 5n² + 5n + 5، فتبقى المتراجحة صحيحة لكل n ≥ 1.

📈

الرسم البياني — تصور المنحنيين

f(n) = 5n² + 5n + 5  vs  C · n² = 15n²

اسحب المنزلق لتغيير C وشاهد كيف يتحرك المنحنى الأخضر. عند C = 15 (قيمة الإثبات)، يتلامسان عند n₀ = 1 ويبقى الأخضر دائمًا أعلى.

f(n) = 5n² + 5n + 5
C · n² (الحد العلوي)
n₀ = 1 (نقطة البداية)
C = 15
n_max = 7

عرض مقرّب — حول n₀ = 1

هنا نقترب لنرى بالضبط أين وكيف يتخطى المنحنى الأخضر المنحنى الأحمر.

📋

جدول القيم — تأكيد المتراجحة

يوضح الجدول أدناه f(n) و 15n² لكل قيمة n. الصف الأخضر هو n₀ = 1 — لاحظ كيف يكونان متساويين عند n = 1، ومن n = 2 فصاعدًا يكون 15n² أكبر بصرامة.

n f(n) = 5n² + 5n + 5 C · n² = 15n² f(n) ≤ 15n²؟ الفرق (15n² − f(n))
✓

الخلاصة — الإثبات الرسمي

∴ إذن — بتعريف رمز O الكبير
Since  5n² + 5n + 5  ≤  15 · n²  for all n ≥ 1,

with  C = 15  and  n₀ = 1 ,

we conclude that  5n² + 5n + 5  is  O(n²).

الملخص — ما فعلناه في 3 خطوات

1
وجدنا C: استبدلنا كل حد غير مهيمن بـ شيء · n² للحصول على الحد العلوي 15n². إذن C = 15.
2
وجدنا n₀: عوّضنا n = 1 وتحققنا من أن المتراجحة 15 ≤ 15 صحيحة. إذن n₀ = 1.
3
كتبنا الاستنتاج: بتعريف رمز O الكبير، بما أننا وجدنا C و n₀، فإن 5n² + 5n + 5 = O(n²) ثابتة. ✓

🎯 هل O(n²) أضيق حد ممكن؟ نعم! الحد المهيمن في 5n² + 5n + 5 هو 5n². نحذف الثوابت والحدود الأدنى درجة → O(n²). هذا أضيق فئة Big O ممكنة.

يمكننا أيضًا القول 5n² + 5n + 5 = O(n³) أو O(n²²)، لكنها حدود غير دقيقة. الأدق هو O(n²).
