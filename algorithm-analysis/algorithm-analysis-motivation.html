1 The Challenge — Ahmed vs. Omar

Two students. Two algorithms. One question.

📚 Imagine two computer science students, Ahmed and Omar, are competing in a class assignment. They each write their own sorting algorithm to sort an array of 1 million numbers. Both are confident their algorithm is the best.

After testing, they come back to class with their results…

🧑
Ahmed
My sorting algorithm sorted 1 million numbers in just 5 seconds. I am clearly the winner!
⏱ 5 seconds
🧑
Omar
My algorithm took 14 seconds for the same task — but wait, before you judge, let me explain!
⏱ 14 seconds
❓ The Big Question
Is Ahmed's algorithm truly faster than Omar's?
Think carefully — 5 seconds is less than 14 seconds, so Ahmed wins… right?

⚠️ Not so fast! The answer is not obvious. Before we can judge, we need to ask a few very important questions. The race is not always fair — and timing alone does not tell the whole story.

2 Imagine a Running Race…

A simple story to understand unfair comparisons

Imagine three runners are competing in a race. Each runner covers the same distance, but they are running on completely different surfaces:

  • Runner 1 runs on solid pavement — fast and smooth
  • Runner 2 runs through deep mud — every step is a struggle
  • Runner 3 runs on loose sand — slow and difficult

Runner 1 finishes in 10 seconds. Runner 2 finishes in 45 seconds. Runner 3 finishes in 28 seconds.

The comparison is unfair because the conditions are different. The same problem exists when comparing Ahmed and Omar.

3 Back to Ahmed & Omar — Why Is It Unfair?

Three reasons why raw timing is not a reliable judge

Now think about Ahmed and Omar. What if one of these situations is true?

💻

Different Computers

Ahmed might be running his algorithm on a brand new, high-performance workstation, while Omar is using an old, slow laptop. Ahmed's computer is the "solid pavement" — of course it is faster!

👨‍💻

Different Programmers

Maybe Omar's algorithm is actually more efficient, but his Java implementation is poorly written. Ahmed is a more experienced coder and wrote clean, optimized code, so his version runs faster.

🔨

Different Tools

Even on the same computer, different programming languages, compilers, or runtime environments can cause significant time differences — even for the exact same algorithm logic.

💡 The Real Problem
How do we fairly judge which algorithm is better?
We need a measurement that is independent of all these external factors

🔎 Solution idea: Have the same professional programmer implement both algorithms and run them on the same computer. This would be fairer — but it is still not the best approach. We need something even more fundamental and universally fair.

4 What We Need — The Fair Judge

A measurement that is truly universal and objective

To fairly measure and compare algorithms, we need a method that is completely independent of all external factors:

💻
Independent of the Computer Whether you run it on an old laptop or a supercomputer, the judgment should be the same
👨‍💻
Independent of the Programmer The skill or experience of the person who writes the code should not affect the result
🔨
Independent of the Programming Language Java, Python, C++, or any other language should not change the fundamental measurement
📊
Based on How the Algorithm Behaves as Input Grows What happens when n doubles? When n is 1 million vs 1 billion? This is the true measure of an algorithm
🌟

The Answer: Algorithm Analysis & Big O Notation

We study how many operations an algorithm performs as a function of its input size n. This gives us a mathematical, computer-independent, programmer-independent measure of an algorithm's true efficiency.

5 The Building Block — What Is O(1)?

The smallest unit of work in any algorithm

To count how many operations an algorithm performs, we first need to agree on what counts as one operation. In computer science, we define any simple, single statement as taking one unit of time — we call this O(1), or constant time.

💡 Important agreement: We cannot measure the exact time of a statement (it changes with every computer). So instead, we agree: every single statement, no matter what computer runs it, counts as 1 operation = O(1). This is our fair, universal unit.

Examples of single statements — each one is O(1):

i = i + 1;
→
O(1)
1 operation
Even though the CPU translates i = i + 1 into several micro-instructions internally, we agree to count it as one operation. This makes our analysis independent of the hardware.
x = arr[5];
→
O(1)
1 operation
Array access by index is always one operation, whether the array has 10 elements or 10 million. It takes the same constant effort.
sum = a + b;
→
O(1)
1 operation
Arithmetic operations, variable assignments, comparisons — all are O(1). The result does not depend on any input size n.

✅ Key insight: O(1) does not mean "takes 1 second" or "takes 1 millisecond." It means "takes a constant amount of time — the same amount regardless of the problem size." This is our universal, fair unit of measurement.

6 From O(1) to O(n) — How Loops Work

Building up from single operations to algorithms

Now that we know O(1) is our building block, let us see what happens when we put that operation inside a loop. This is how we start to count the total work an algorithm does.

int x = 0; x = x + 1; // ← 1 statement
Total operations:
O(1)
= O(1)
for (int i = 0; i < 10; i++) { x = x + 1; // executes 10 times }
One O(1) operation repeated 10 times:
O(1) O(1) O(1) O(1) O(1) O(1) O(1) O(1) O(1) O(1)
O(1) × 10 = O(10)
for (int i = 0; i < n; i++) { x = x + 1; // executes n times }
One O(1) operation repeated n times:
O(1) O(1) O(1) … O(1)
O(1) × n = O(n)

💡 The pattern: A loop that runs n times, with O(1) work inside, gives us O(1) + O(1) + … + O(1) (n times) = O(n). The number of operations grows linearly with the input size. Double the input → double the work.

What about nested loops?

for (int i = 0; i < n; i++) { // n times for (int j = 0; j < n; j++) { // n times x = x + 1; // O(1) } }
Inner loop runs n times for each of the n outer iterations:
n × n
= n² operations = O(n²)

✅ The big idea: We do not need to know the exact running time in seconds. We count the number of operations as a mathematical function of the input size n. This count is the same regardless of the computer, the programmer, or the language. That is our fair judge.

7 Algorithm Behaviors — The Nine Complexity Classes

How different algorithms grow as input size increases

We do not need to know the exact number of operations. What we care about is the behavior pattern — how the operation count grows as input size n gets larger. We call this the Big O notation. Every algorithm falls into one of these nine complexity classes. Explore each one below to see a real graph and data table showing exactly how fast each complexity grows.

O(1) — Constant O(log n) — Logarithmic O(√n) — Square Root O(n) — Linear O(n·log n) — Linearithmic O(n²) — Quadratic O(n³) — Cubic O(2ⁿ) — Exponential O(n!) — Factorial O(nⁿ) — n-to-the-n
O(1) Constant Time

What it means: The algorithm always takes the same number of operations, no matter how large the input is. The problem size simply does not affect the running time.

Real example: Accessing an element by index — arr[5] is just as fast whether the array has 10 or 10 million elements.

Notice in the graph: A perfectly flat horizontal line. As n grows from 1 to 100, the operations stay at exactly 1.

Java — O(1) Constant Time ▼
// O(1) — Constant Time: operations do NOT depend on n int getFirst(int[] arr) { return arr[0]; // Always exactly 1 operation }

f(n) = 1  —  Operations stay the same regardless of input size

Input Size (n) Operations f(n) = 1
O(log n) Logarithmic Time

What it means: Each time n doubles, the number of operations increases by only 1. It grows, but extremely slowly.

Real example: Binary search — searching a sorted array of 1 million elements takes only about 20 comparisons!

Notice in the graph: The curve rises steeply at first, then bends and almost flattens. Even for 100 million elements, log₂(n) only reaches about 27 operations.

Java — O(log n) Logarithmic Time ▼
// O(log n) — Logarithmic: n is halved each iteration void logExample(int n) { while (n > 1) { n = n / 2; // n is cut in half → only log₂(n) iterations needed } }

f(n) = log₂(n)  —  Graph: n = 1 to 1,000  |  Table: up to 100 million

Input Size (n) Operations f(n) = log₂(n)
O(√n) Square Root Time

What it means: Operations grow as the square root of input size — much slower than linear. When n quadruples, operations only double.

Real example: Primality testing — to check if n is prime you only need to test divisors up to √n, not all the way to n.

Notice in the graph: The curve rises quickly at first then gradually flattens. For n = 10,000, only 100 operations are needed instead of 10,000.

Java — O(√n) Square Root Time ▼
// O(√n) — Square Root: loop runs only up to √n boolean isPrime(int n) { if (n < 2) return false; for (int i = 2; i * i <= n; i++) { // loop runs √n times if (n % i == 0) return false; } return true; } // Condition i*i ≤ n stops the loop at exactly √n iterations

f(n) = √n  —  Grows much slower than linear

Input Size (n) Operations f(n) = √n
O(n) Linear Time

What it means: Operations grow proportionally with input size. If n doubles, operations double too. A consistent, steady increase.

Real example: Linear search — scanning an unsorted list element by element from start to end.

Notice in the graph: A perfectly straight diagonal line. Every increase in n adds the same fixed number of operations.

Java — O(n) Linear Time ▼
// O(n) — Linear: one pass through all n elements int linearSearch(int[] arr, int target) { for (int i = 0; i < arr.length; i++) { // runs n times if (arr[i] == target) return i; } return -1; }

f(n) = n  —  Operations grow in a perfectly straight line

Input Size (n) Operations f(n) = n
O(n·log n) Linearithmic Time

What it means: Operations grow slightly faster than linear but far slower than quadratic. When n doubles, operations slightly more than double.

Real example: Merge sort, heap sort — divide into O(log n) levels and do O(n) work at each level, giving O(n·log n) total.

Notice in the graph: Looks nearly linear at small n, then curves gently upward. This is the best possible speed for comparison-based sorting.

Java — O(n·log n) Linearithmic Time ▼
// O(n·log n) — the loop pattern seen in merge sort // outer loop: n iterations; inner loop halves j → log₂(n) iterations void nLogNExample(int n) { for (int i = 1; i <= n; i++) { // runs n times for (int j = n; j >= 1; j = j / 2) { // halves → log₂(n) iterations process(i, j); } } } // Total: n × log₂(n) operations

f(n) = n · log₂(n)  —  Slightly above linear, the "sweet spot" for sorting

Input Size (n) Operations f(n) = n · log₂(n)
O(n²) Quadratic Time

What it means: Operations grow as the square of input size. If n doubles, operations multiply by 4!

Real example: Bubble sort, selection sort — two nested loops each running n times, giving n × n = n² operations.

Notice in the graph: The curve bends sharply upward (a parabola). For n=100, we already have 10,000 operations — compare that to just 100 for O(n).

Java — O(n²) Quadratic Time ▼
// O(n²) — Quadratic: two nested loops void printPairs(int[] arr, int n) { for (int i = 0; i < n; i++) { // outer loop: n times for (int j = 0; j < n; j++) { // inner loop: n times process(arr[i], arr[j]); } } } // Total: n × n = n² operations

f(n) = n²  —  Parabola: grows much faster than linear

Input Size (n) Operations f(n) = n²
O(n³) Cubic Time

What it means: Operations grow as the cube of input size. Even more explosive than quadratic. If n doubles, operations multiply by 8!

Real example: Naive matrix multiplication — three nested loops each running n times.

Notice in the graph: The curve rises even more steeply than O(n²). For n=20, we have 8,000 operations. For n=50, we have 125,000 — notice how fast it explodes.

Java — O(n³) Cubic Time ▼
// O(n³) — Cubic: three nested loops void tripleLoop(int n) { for (int i = 0; i < n; i++) { // n times for (int j = 0; j < n; j++) { // n times for (int k = 0; k < n; k++) { // n times process(i, j, k); } } } } // Total: n × n × n = n³ operations

f(n) = n³  —  Steeper than quadratic, grows very fast

Input Size (n) Operations f(n) = n³
O(2ⁿ) Exponential Time

What it means: Operations double with every single increase in n. This is the most explosive growth possible. Every +1 to n multiplies the operations by 2.

Real example: Recursive Fibonacci without memoization, generating all subsets of a set, brute-force password cracking.

Notice in the graph: The curve appears almost flat, then shoots straight up like a wall. At n=30, we already have over 1 billion operations — this is why exponential algorithms are impractical for any meaningful input size.

Java — O(2ⁿ) Exponential Time ▼
// O(2ⁿ) — Exponential: two recursive calls each time int fibonacci(int n) { if (n <= 1) return n; // base case return fibonacci(n-1) + fibonacci(n-2); // 2 calls → branches double every level → 2ⁿ }

f(n) = 2ⁿ  —  Operations double with every +1 to n

Input Size (n) Operations f(n) = 2ⁿ
O(n!) Factorial Time

What it means: Operations grow as n factorial: n × (n−1) × … × 1. Growth is catastrophically fast — at n=20 that is already 2.4 quadrillion operations.

Real example: Brute-force permutation — generating every possible ordering of n items (e.g., the Travelling Salesman Problem solved naively).

Notice in the graph: Even for tiny n (up to 10), the value explodes dramatically. Factorial algorithms are only practical for very small inputs.

Java — O(n!) Factorial Time ▼
// O(n!) — Factorial: generating all permutations void permute(int[] arr, int start) { if (start == arr.length) { process(arr); // one complete permutation return; } for (int i = start; i < arr.length; i++) { swap(arr, start, i); // try each element permute(arr, start + 1); // recurse on remaining swap(arr, start, i); // backtrack } } // Total recursive calls: n! — one per permutation

f(n) = n!  —  Catastrophic growth, only practical for very small n

Input Size (n) Operations f(n) = n!
O(nn) n-to-the-n Time

What it means: The theoretical worst-case complexity class. Operations equal n multiplied by itself n times. It grows faster than factorial — n! < nn for all n > 1.

Real example: Exhaustive enumeration where each of n positions can independently take any of n values. Mostly a theoretical upper bound, rarely seen in practice.

Notice in the graph: Even at n=8, the value exceeds 16 million. This class defines the mathematical ceiling for complexity analysis.

Java — O(nn) Concept ▼
// O(nⁿ) — each of n positions can independently take any of n values // Example: enumerate all n-digit strings over an alphabet of size n void enumerateAll(int n, int[] curr, int pos) { if (pos == n) { process(curr); // one complete assignment return; } for (int v = 0; v < n; v++) { // n choices at each position curr[pos] = v; enumerateAll(n, curr, pos + 1); } } // Total: n × n × … × n (n times) = nⁿ

f(n) = nn  —  Theoretical worst case, grows faster than factorial

Input Size (n) Operations f(n) = nn

👀 Let us work through a complete, concrete example from start to finish. Below is a Java method called counter(n). It uses three separate loop sections, each performing a different number of operations depending on n. Before analyzing it, let us first run it and see the output.

Java — PolynomialCounter.java ▼
public class PolynomialCounter { public static int counter(int n) { int c = 0; // Section A — 5n² for (int a = 1; a <= 5; a++) { for (int i = 1; i <= n; i++) { for (int j = 1; j <= n; j++) { c++; } } } // Section B — 5n for (int a = 1; a <= 5; a++) { for (int i = 1; i <= n; i++) { c++; } } // Section C — 5 (constant) for (int i = 1; i <= 5; i++) { c++; } return c; } public static void main(String[] args) { System.out.println(counter(5)); // What does this print? } }

👀 When n = 5, what does counter(5) print?

counter(5) → 155 operations counted
💡 The Key Insight
Can we predict 155 without running the code?
YES — by expressing the algorithm as a polynomial function of n

Each section of counter(n) contributes exactly one term to the polynomial:

Section A — 3 nested loops
Outer: a = 1 to 5 (fixed 5 times)
Inner: i = 1 to n, j = 1 to n
→ 5 × n × n = 5n² increments
5n²
Section B — 2 nested loops
Outer: a = 1 to 5 (fixed 5 times)
Inner: i = 1 to n
→ 5 × n = 5n increments
5n
Section C — 1 loop (constant)
i = 1 to 5 (never depends on n)
→ always exactly 5 increments
5
Total operations in counter(n) = sum of all three sections
f(n) = 5n² + 5n + 5
✅ Proof: substitute n = 5 into the polynomial
f(5) = 5(5²) + 5(5) + 5 = 5(25) + 25 + 5 = 125 + 25 + 5 = 155
🎉 Exactly matches the output! The polynomial is the algorithm.

📌 But wait — a mathematician would ask:
How do you know that's valid?  Prove it.

We just concluded counter(n) = O(n²) by intuition — keep the biggest term, drop everything else. That is a useful shortcut, but it is not a proof.

To make it rigorous, we must find concrete values of C and n₀ and show the inequality holds. That is exactly what we do next.

💡

What Does "O(n²)" Mean Visually?

When we say f(n) is O(g(n)), we mean we can find a multiple of g(n) that eventually grows faster than f(n) and stays above it forever after some point n₀.

In our case:

  • Red curve = our function f(n) = 5n² + 5n + 5
  • Green curve = our bound C · n² — we need to find a C that stays above the red curve
  • n₀ = the point after which the green curve is always ≥ the red curve

The two curves may intersect at n₀ — what matters is that for all n ≥ n₀, the green curve dominates.

📖

Formal Definition of Big O

Official Definition
f(n) is O(g(n))   ⟺   ∃ positive constants C and n₀ such that:
f(n) ≤ C · g(n)    for all n ≥ n₀
C
A positive constant multiplier we must find
n₀
The starting point from which the bound holds
f(n) ≤ C·g(n)
The inequality that must be satisfied

Our goal: Show that 5n² + 5n + 5 ≤ C · n² for some positive C and for all n ≥ n₀. We need to find the exact values of C and n₀.

1

Step 1 — Find the Constant C

Strategy (the standard method): Replace every term in f(n) with something larger that uses only n². This gives us a clean upper bound of the form C · n².

f(n)  =  5n²  +  5n  +  5 original function
≤ 5n²  +  5n²  +  5n² 5n ≤ 5n² and 5 ≤ 5n²  for n ≥ 1
= (5 + 5 + 5) · n²  =  15n² collect coefficients
Why is 5n ≤ 5n²? Because for any n ≥ 1, we have n ≤ n², so 5n ≤ 5n².  ✓
Why is 5 ≤ 5n²? Because for any n ≥ 1, we have 1 ≤ n², so 5 ≤ 5n².  ✓
🎯
We found C
C = 15   →   5n² + 5n + 5 ≤ 15n²
2

Step 2 — Find n₀ (Verify the Inequality)

Strategy: We always start by trying n = 1. If the inequality f(1) ≤ C · g(1) is true, then n₀ = 1. If not, try n = 2, 3, … until it holds.

Try n = 1:
LHS: 5(1²) + 5(1) + 5  =  5 + 5 + 5  =  15
RHS: 15 · (1²)  =  15
⇒ 15 ≤ 15 ✓ TRUE (equality holds!)
🔎 Verification at n = 1
f(1) = 5·1² + 5·1 + 5 = 5 + 5 + 5 = 15
C·g(1) = 15·1² = 15

15 ≤ 15 → TRUE ✓

The inequality holds at n = 1, so n₀ = 1.

📍
We found n₀
n₀ = 1   →   The bound holds for all n ≥ 1

💡 Note: At n = 1 the two curves are equal (both = 15). For all n > 1, the bound C·n² = 15n² grows strictly faster than 5n² + 5n + 5, so the inequality holds for all n ≥ 1.

Conclusion — The Formal Proof

∴ Therefore — By definition of Big O
Since  5n² + 5n + 5  ≤  15 · n²  for all n ≥ 1,

with  C = 15  and  n₀ = 1 ,

we conclude that  5n² + 5n + 5  is  O(n²).

Summary — What we did in 3 steps

1
Found C: Replaced each non-dominant term with something · n² to get an upper bound of 15n². So C = 15.
2
Found n₀: Plugged in n = 1 and verified the inequality 15 ≤ 15 is true. So n₀ = 1.
3
Wrote the conclusion: By definition of Big O, since we found C and n₀, 5n² + 5n + 5 = O(n²) is proven. ✓

🎯 Is O(n²) the tightest bound? Yes! The dominant term of 5n² + 5n + 5 is 5n². We drop constants and lower-order terms → O(n²). That is the tightest possible Big O class.

We could also say 5n² + 5n + 5 = O(n³) or O(n²²), but those are loose bounds. The tightest is O(n²).
