1 · Why Do We Need O(n log n)?

In the previous chapter we studied quadratic sorting algorithms — Selection Sort, Insertion Sort, and Bubble Sort. They are simple and educational, but their O(n²) complexity makes them impractical for large datasets.

O(n²)
Quadratic

n = 1,000,000 → 1012 operations
Too slow for real-world data

O(n log n)
Logarithmic

n = 1,000,000 → ~20,000,000 operations
50,000× faster than O(n²)!

Key Question: Can we sort faster than O(n²)?
Answer: Yes — by using Divide and Conquer. Instead of comparing every element with every other element, we divide the problem in half at each step, solve each half, then combine the results. This reduces the number of levels to log₂(n).

The Divide and Conquer Strategy

  1. Divide — Split the problem into smaller sub-problems
  2. Conquer — Solve each sub-problem recursively
  3. Combine — Merge the solved sub-problems into a final solution
Why log n levels? If we halve the array at each step, starting from n elements: n → n/2 → n/4 → n/8 → … → 1. The number of times we can halve n before reaching 1 is log₂(n). At n = 1,000,000, that is only about 20 levels.

2 · Chapter Overview

We study two O(n log n) sorting algorithms in this chapter, both based on Divide and Conquer:

Quick Sort

  • Choose a pivot element
  • Partition: elements < pivot go left, elements > pivot go right
  • Recursively sort each partition
  • Average O(n log n), Worst O(n²)
  • In-place (no extra array needed)

Merge Sort

  • Divide array in half
  • Recursively sort each half
  • Merge two sorted halves into one
  • Always O(n log n) — guaranteed
  • Requires O(n) extra space

The Recurrence — Why Both Are O(n log n)

Both algorithms share the same recurrence relation in their average/typical case:

T(n) = 2 · T(n2) + n

Solving by substitution (or Master Theorem):

T(n) = n log₂ n  ∈  O(n log n)

At each level: O(n) work. Number of levels: log₂(n). Total: O(n log n).

3 · Quick Sort

Classroom Motivation — The Story Behind Quick Sort

Imagine this classroom scenario...

Your teacher wants to arrange 8 students in a single line from shortest to tallest. Each student holds a phone that displays their exact height. They are currently standing in a random order. How should the teacher sort them efficiently?

1
Naive Approach: Compare Every Student with Every Other — O(n²)

Take Ali (155 cm). Compare him with Sara (162 cm) — Sara is taller. Compare Ali with Fahad (178 cm) — Fahad is taller. And so on for every pair. You must compare every student with every other student.

For 8 students:  8 × 7 / 2 = 28 comparisons.
For 1,000 students:  499,500 comparisons.
For 1,000,000 students:  500 billion comparisons!  → O(n²)

2
The Smart Idea: Pick One Student as a "Divider" (Pivot)

Pick Sara (162 cm) — the first student in line — and call her the pivot. Now ask each of the other 7 students one simple question:

"Are you shorter than Sara (162 cm)? → Step to the LEFT."
"Are you taller than Sara (162 cm)? → Step to the RIGHT."

Only 7 comparisons to split the entire group. And Sara is now in her correct final position — permanently. She never needs to move again.

3
Now Sort Each Group Independently — Divide and Conquer!

The left group (4 students) and the right group (3 students) are completely independent — they will never need to compare with each other again, because we already know every student on the left is shorter than every student on the right.

Apply the exact same pivot idea recursively inside each group: pick a pivot, ask everyone to go left or right, repeat. Each level of recursion does at most n comparisons in total across all groups at that level. With balanced splits there are only log₂(8) = 3 levels:

3 levels × 8 comparisons per level = 24 comparisons  vs  28 for the naive approach  →  O(n log n)

For 1,000,000 students: only ~20,000,000 comparisons instead of 500 billion. That is 25,000× faster!

!
The Big Challenge: What if You Pick the Wrong Pivot?

What if, instead of Sara (162 cm), we had picked Hana (145 cm) — the shortest student? All 7 remaining students are taller → they all go to the right. The left side is empty. We only shrank the problem by 1 student. This repeats n times:

n levels × n comparisons per level = O(n²) — same as the naive approach!

Choosing a good pivot is the central challenge of Quick Sort. How do you reliably find the "middle" student? This is an active research problem in computer science — it is called the pivot selection problem and is still studied today.

This is exactly what Quick Sort does! It picks a pivot element, partitions everything smaller to the left and everything larger to the right in a single pass, then recursively repeats on each partition. Despite the worst-case risk, Quick Sort is the most widely used sorting algorithm in practice due to its outstanding speed and cache performance.
Core Idea: Choose a pivot element. Rearrange (partition) the array so everything smaller than the pivot is on the left and everything larger is on the right. The pivot is now in its correct final position. Recursively apply the same process to each half.

How It Works — Hoare Partition Scheme

  1. Choose a pivot element. The pivot can be selected randomly, or using other strategies (first, last, median-of-three). In this tutorial we use the first element A[l] as the pivot.
  2. Partition the array around the pivot:
    • Pointer i scans from the left, looking for elements greater than the pivot.
    • Pointer j scans from the right, looking for elements smaller or equal to the pivot.
    • When both stop, swap A[i] and A[j].
    • When i ≥ j, place the pivot at position j — it is now in its final sorted position.
  3. Recursively sort the left sub-array (indices l … j−1) and the right sub-array (j+1 … h).
  4. Base case: a sub-array of size 0 or 1 is already sorted — stop.

Algorithm Analysis

Average Case and Best Case — O(n log n):

Each partition splits the array roughly in half → log₂(n) levels × O(n) work per level

T(n) = 2 · T(n2) + n   →   O(n log n)

Worst Case — O(n²):

Occurs when the pivot is always the smallest or largest element (sorted or reverse-sorted input).

Each partition produces one sub-array of size n−1 and one of size 0 → n levels → O(n) work each

T(n) = T(n−1) + n   →   O(n²)

Avoiding the worst case: Choose a random pivot, or use the median-of-three strategy (take the median of first, middle, and last elements). In practice, Quick Sort is often the fastest algorithm due to excellent cache performance.

Java Implementation

Quick Sort — Hoare Partition Scheme (Java)▼ 
// ═══════════════════════════════════════════════════════════════════════
//  PART 1 — quickSort   (the main recursive function)
//  Goal : sort the sub-array  A[l .. h]  in ascending order.
// ═══════════════════════════════════════════════════════════════════════
public static void quickSort(int[] A, int l, int h) {

    // Base case: if the sub-array has 0 or 1 element it is already sorted → stop.
    if (l < h) {

        // Partition the array and receive the pivot's final sorted position j.
        int j = partition(A, l, h);

        // Recursively sort the LEFT  partition → all elements smaller than the pivot.
        quickSort(A, l, j - 1);

        // Recursively sort the RIGHT partition → all elements greater than the pivot.
        quickSort(A, j + 1, h);
    }
}


// ═══════════════════════════════════════════════════════════════════════
//  PART 2 — partition   (Hoare Partition Scheme)
//  Goal : rearrange A[l..h] so that
//           A[l .. j]    contains elements  ≤ pivot   (left partition)
//           A[j+1 .. h]  contains elements  > pivot   (right partition)
//  Returns j — the pivot's final sorted index.
// ═══════════════════════════════════════════════════════════════════════
public static int partition(int[] A, int l, int h) {

    // ── Choose the pivot ─────────────────────────────────────────────────
    // Here we use the FIRST element A[l] as the pivot.
    // The pivot can also be chosen randomly or by other strategies.
    int pivot = A[l];

    // ── Set up the two pointers ──────────────────────────────────────────
    // i starts ONE STEP BEFORE the left boundary  → will scan to the RIGHT →
    // j starts ONE STEP AFTER  the right boundary → will scan to the LEFT  ←
    // Starting outside the range lets the do-while loops enter correctly.
    int i = l - 1;
    int j = h + 1;


    // ── Outer loop — repeat until i and j cross each other ───────────────
    while (i < j) {

        // ── MOVE i TO THE RIGHT  → ───────────────────────────────────────
        // Advance i one step at a time.
        //   • While A[i] <= pivot : this element is small enough — it BELONGS
        //     on the left side — keep moving right.
        //   • Stop  when  A[i]  > pivot : this element is TOO BIG for the left
        //     side — it is in the WRONG place and needs to move right.
        do {
            i++;
        } while (A[i] <= pivot);

        // ── MOVE j TO THE LEFT   ← ───────────────────────────────────────
        // Advance j one step at a time (moving leftward).
        //   • While A[j]  > pivot : this element is large enough — it BELONGS
        //     on the right side — keep moving left.
        //   • Stop  when  A[j] <= pivot : this element is TOO SMALL for the right
        //     side — it is in the WRONG place and needs to move left.
        do {
            j--;
        } while (A[j] > pivot);

        // ── SWAP if i has NOT yet crossed j ──────────────────────────────
        // Both pointers stopped on misplaced elements:
        //   A[i] is too BIG  (> pivot) but sits on the LEFT  side — wrong place.
        //   A[j] is too SMALL (≤ pivot) but sits on the RIGHT side — wrong place.
        // Swapping them corrects both elements in one move.
        //
        // If i >= j the pointers have CROSSED — partitioning is complete.
        // Do NOT swap — the loop condition (i < j) will exit on the next check.
        if (i < j) {
            swap(A, i, j);
        }

    } // ── end of outer while loop ──────────────────────────────────────────


    // ── Place the pivot in its final sorted position ─────────────────────
    // After the loop ends, j marks the boundary between the two partitions:
    //     A[l .. j]    → all ≤ pivot   (left partition)
    //     A[j+1 .. h]  → all  > pivot  (right partition)
    //
    // The pivot is still at A[l] (its original position).
    // Swap A[l] with A[j] to move the pivot to index j — the centre.
    // The pivot is now in its CORRECT FINAL SORTED POSITION.
    swap(A, l, j);

    // Return j so quickSort knows where to split for the two recursive calls.
    return j;
}


// ═══════════════════════════════════════════════════════════════════════
//  PART 3 — swap helper
//  Exchanges the values at positions i and j inside the array.
// ═══════════════════════════════════════════════════════════════════════
private static void swap(int[] A, int i, int j) {
    int temp = A[i];    // Step 1 — save A[i] in a temporary variable so we don't lose it
    A[i]      = A[j];   // Step 2 — overwrite A[i] with the value from A[j]
    A[j]      = temp;   // Step 3 — put the saved original value of A[i] into A[j]
}

Understanding the Code — Step by Step

P
Choosing the Pivot

pivot = A[l] — we pick the first element of the current sub-array as the pivot value. The pivot is the element that will end up in its correct final sorted position after one call to partition(). The pivot value can also be chosen randomly — the two-pointer mechanism works the same way regardless of which element is chosen.

i →
Moving pointer i to the RIGHT

i starts at l − 1 (one step before the left boundary). The do { i++; } while (A[i] <= pivot) loop advances it rightward:

  • Keep going while A[i] ≤ pivot → the element is small enough, it belongs on the left.
  • Stop when A[i] > pivot → found an element that is too big for the left side — it needs to swap to the right.
← j
Moving pointer j to the LEFT

j starts at h + 1 (one step after the right boundary). The do { j--; } while (A[j] > pivot) loop advances it leftward:

  • Keep going while A[j] > pivot → the element is large enough, it belongs on the right.
  • Stop when A[j] ≤ pivot → found an element that is too small for the right side — it needs to swap to the left.
↔
Swapping the two misplaced elements

Both pointers stopped on elements that are in the wrong place. If i < j (not yet crossed), call swap(A, i, j):

  • A[i] was too big on the left side → moves to the right ✓
  • A[j] was too small on the right side → moves to the left ✓

After the swap, the while (i < j) outer loop repeats — i and j continue scanning from where they stopped until they find the next pair of misplaced elements.

i≥j
When i ≥ j — pointers have crossed → loop exits

When i reaches or passes j, it means every element has been checked:

  • The if (i < j) condition is false → no swap happens.
  • The outer while (i < j) condition is also false → the loop exits.

The array is now partitioned: A[l..j] holds all elements ≤ pivot, and A[j+1..h] holds all elements > pivot. Finally swap(A, l, j) moves the pivot from A[l] into A[j] — its correct sorted position forever.

4 · Merge Sort

Core Idea: Divide the array in half, recursively sort each half, then merge the two sorted halves into one sorted array. Unlike Quick Sort, no pivot or partitioning is needed — the "work" happens entirely during the merge step.

Why Merge Sort Matters — The Big Data Problem

A real-world challenge the professor poses to the class…

Your company needs to sort a log file of 10 GB, but the server has only 1–2 GB of RAM. How do you sort a dataset that is 5× larger than your available memory?

1
Divide the 10 GB file into 5 chunks of 2 GB each

Load each 2 GB chunk into RAM one at a time, sort it completely (using Quick Sort, for example — it fits in memory now!), then write the sorted result back to disk. Repeat for all 5 chunks. You now have 5 sorted files on disk.

Open all 5 sorted files simultaneously. Read just one record from each. You now have 5 values in memory — pick the smallest, write it to the output file on disk, then read the next record from the file that just contributed. Repeat until all files are exhausted.

Memory used during the entire merge: only 5 records at a time — no matter how many gigabytes the files are!

?
Student: "Wait — to join two 2 GB sorted chunks, don't we need 4 GB of RAM?"

Great question! The answer is no — and this is the key insight of Merge Sort. You do not need both files in memory at the same time. You open both files, read one element from each, compare them, write the smaller to the output file on disk, then read the next element from whichever file just contributed. At any moment you hold exactly one element from each file in memory.

The merged result is written directly to disk, not to a RAM buffer. This is called external sorting — and Merge Sort is the algorithm uniquely suited for it. It is used by databases, operating systems, and distributed computing frameworks (like Hadoop and Spark) to sort terabytes of data on clusters of machines.

The key operation in all of this: merging two sorted sequences into one sorted sequence. Once you master this single skill, you understand both the algorithm and its real-world power.

Step 0 — The Core Skill: Merging Two Sorted Arrays

Before studying the full recursive Merge Sort algorithm, let us master the merge step in isolation. This is the "combine" phase of Divide and Conquer — where all the real work of Merge Sort happens.

Problem: Given two sorted arrays, produce one merged sorted array.

Left   L = [3, 9, 27, 56]
Right R = [10, 38, 43, 82]

Expected output: [3, 9, 10, 27, 38, 43, 56, 82]

The Two-Pointer Algorithm

  1. Place pointer i = 0 at the front of Left array L.
  2. Place pointer j = 0 at the front of Right array R.
  3. While both pointers are within their arrays:
    • If L[i] ≤ R[j]: copy L[i] into the result, advance i.
    • If L[i] > R[j]: copy R[j] into the result, advance j.
  4. When one array is exhausted, copy all remaining elements from the other directly to the result.
Why this works: Because both arrays are already sorted, the globally smallest remaining element is always at the front of one of the two arrays — either L[i] or R[j]. We never need to search beyond those two positions. One comparison, one copy, one pointer advance. This gives us an O(n) merge.

How It Works — The Full Merge Sort Algorithm

  1. Divide: Find the midpoint m = (l + h) / 2. Split into A[l…m] and A[m+1…h].
  2. Conquer: Recursively call mergeSort on each half.
  3. Base case: A sub-array of size 1 (l == h) is already sorted — return immediately.
  4. Merge: Use two pointers i (left half) and j (right half). Compare A[i] and A[j], copy the smaller into a temporary array B, advance that pointer. When one half is exhausted, copy the remainder of the other half. Finally copy B back into A.

Algorithm Analysis

All Cases (Best, Average, Worst) — O(n log n):

Merge Sort always divides exactly in half and always merges all n elements.

There is no worst case — the behaviour is the same regardless of input order.

T(n) = 2 · T(n2) + n   →   O(n log n)

Space Complexity — O(n):

The merge step requires a temporary array B of size n. This is the main trade-off compared to Quick Sort (which is in-place).

Key advantage: Merge Sort is stable (equal elements keep their original order) and guarantees O(n log n) even on sorted, reverse-sorted, or adversarial input. It is the preferred choice when guaranteed performance and stability are required.

Java Implementation

Merge Sort — Java▼ 
// Main recursive function
public static void mergeSort(int[] A, int l, int h) {
    if (l < h) {
        int m = (l + h) / 2;          // find midpoint
        mergeSort(A, l, m);            // sort left half
        mergeSort(A, m + 1, h);        // sort right half
        merge(A, l, m, h);             // merge sorted halves
    }
}

// Merge two sorted sub-arrays A[l..m] and A[m+1..h]
public static void merge(int[] A, int l, int m, int h) {
    int i = l, j = m + 1, k = l;
    int[] B = new int[A.length];      // temporary array

    while (i <= m && j <= h) {
        if (A[i] <= A[j]) {
            B[k++] = A[i++];           // take from left half
        } else {
            B[k++] = A[j++];           // take from right half
        }
    }
    while (i <= m)  B[k++] = A[i++];  // copy remaining left
    while (j <= h)  B[k++] = A[j++];  // copy remaining right

    for (int x = l; x <= h; x++) A[x] = B[x]; // copy back
}

5 · Quick Sort vs Merge Sort

Both algorithms are O(n log n) — but they are not identical. Understanding their differences helps you choose the right algorithm for the right situation.
Property Quick Sort Merge Sort
Best Case O(n log n) O(n log n)
Average Case O(n log n) O(n log n)
Worst Case O(n²) O(n log n)
Space Complexity O(log n) — in-place* O(n) — extra array
Stable? No Yes
Cache Performance Excellent Good
Preferred for General-purpose, in-memory sorting Linked lists, external sorting, stable sort required

* O(log n) stack space for recursion

The Full Picture — All Sorting Algorithms

Algorithm Best Average Worst Space Adaptive?
Selection Sort O(n²) O(n²) O(n²) O(1) No
Bubble Sort (optimized) O(n) O(n²) O(n²) O(1) Yes
Insertion Sort O(n) O(n²) O(n²) O(1) Yes
Quick Sort O(n log n) O(n log n) O(n²) O(log n) No

Practical Guidance:
  • For general in-memory sorting → Quick Sort (fastest in practice due to cache locality)
  • For guaranteed O(n log n) or stable sorting → Merge Sort
  • For small arrays (n < 20) or nearly sorted data → Insertion Sort
  • Most language libraries (Java's Arrays.sort, Python's sorted) use hybrid algorithms that combine Quick Sort / Merge Sort with Insertion Sort for small sub-arrays.
