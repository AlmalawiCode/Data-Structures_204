Introduction: Think of a Cabinet

🗄️ Data Structures are Like Cabinets

A helpful way to understand data structures: think of a cabinet. A wardrobe has shelves and hanging space — designed for clothes. A filing cabinet has labeled drawers — designed for documents. A bookshelf has rows — designed for books.

Each cabinet is built for a specific kind of storage. In the same way, the data structures you have already learned — arrays, linked lists, stacks, queues, and trees — are different kinds of "cabinets" for organizing data inside the computer. Each one is built for a specific purpose, with its own strengths and limitations.

Today, we ask: which cabinet do we use for a map? And we add a powerful new structure to our collection: the Graph.

🗺️

A Real-World Challenge: Cities of Saudi Arabia

Study this map carefully — then ask yourself: how would I store this data in a program?

Saudi Arabian cities (vertices) connected by roads (edges). Each connection has a distance in kilometers. Source: CPCS204 Course Material — King Abdulaziz University, FCIT

Look at the map above. It shows major cities of Saudi Arabia — Riyadh, Jeddah, Mecca, Medina, Dammam, Tabuk, and more — each connected to others by roads, with distances marked in kilometers.

The Challenge Write a program to store: all the cities, which cities are directly connected by road, and the distance (in km) between each connected pair. A traveler should be able to ask: "Which cities can I reach directly from Riyadh?" or "What is the road distance from Jeddah to Mecca?"

🤔 Which data structure would you use?

Think carefully — which of your existing "cabinets" fits this problem?

❌ Array
An array stores a sequence: cities[0]="Riyadh", cities[1]="Jeddah"... But how do you encode that Riyadh connects to both Jeddah and Medina and Dammam? An array just stores values in a line — it has no concept of "connection" between elements.
✗ Cannot represent multiple connections between cities
❌ Linked List
A linked list goes: Riyadh → Jeddah → Mecca → null. Each node has one next pointer. But Riyadh connects to many cities at once — Jeddah, Medina, Dammam, and more. A linked list can only link each city to exactly one neighbor.
✗ Each node has only one "next" — cannot fan out
❌ Tree
A tree has a root with children and grandchildren — a hierarchy. But Riyadh and Jeddah are equal neighboring cities, not parent and child. Trees cannot represent cycles (circular routes), and road networks are full of them. The structure is simply wrong for this data.
✗ Hierarchical — cities are peers, not parent/child
✅ The Answer: A Graph!
A Graph is a data structure that stores a set of entities (cities) and the connections (roads) between them — where any entity can connect to any number of other entities, in any direction, with any weight (distance). Riyadh can connect to five cities, Jeddah to four, each road storing its own distance. No other structure you have learned can do this naturally.

This is what we study today. Graphs are one of the most powerful and widely-used structures in all of computer science — and they begin right here, with a map of Saudi Arabia.

Before we define it formally, consider these everyday systems you already use — all of them are graphs:

🌍

Google Maps

Cities and locations are connected by roads. Finding the shortest route between two cities is a graph problem solved millions of times per second.

👥

Social Networks

Facebook, LinkedIn, and Twitter are graphs. People are nodes, friendships and follows are edges. "Friends of friends" is graph traversal.

✈

Flight Routes

Airlines connect airports with direct flights. Planning a trip with a layover is finding a path through a graph of airports and routes.

🌐

Internet Routing

Computers and servers are connected by cables and wireless links. Every webpage you load travels through a graph of network devices.

⚙

Task Scheduling

In compilers and build systems, tasks have dependencies — "task A must run before task B." This is a directed graph called a DAG.

Key Insight All of these problems share a common structure: a set of things (cities, people, computers, tasks) connected by relationships (roads, friendships, cables, dependencies). Graphs are the data structure that captures exactly this — many-to-many connections.

What Is a Graph? — Formal Definition

A graph is defined mathematically as an ordered pair:

G = (V, E) Where V is a set of vertices (also called nodes) and E is a set of edges (also called connections or links). Each edge connects two vertices.

For example, a graph with five cities and four connecting roads would have V = {A, B, C, D, E} and E = {(A,B), (A,C), (B,D), (C,D), (D,E)}.

In this graph, V = {A, B, C, D, E} — five vertices. The edges E = {(A,B), (A,C), (B,D), (C,D), (D,E)} connect them. Notice that this graph has no direction (edges go both ways) and no weights — it is the simplest kind.

Graphs vs Trees A tree is actually a special type of graph — a connected, undirected graph with no cycles. Graphs are more general: they can have cycles, disconnected components, directed edges, and weighted edges.

Why Graphs?

You might ask: "Can't we store graph data in an array or linked list?" Let us think carefully about what arrays and linked lists can and cannot represent.

Problem: Arrays and Linked Lists are Linear An array stores a sequence: element[0], element[1], element[2], ... A linked list stores a chain: node → node → node → null. Both structures have one fundamental property: each element has at most one predecessor and one successor. This makes them linear — fine for sequences, but broken for networks.

Consider a road network. The city of Jeddah is connected to Mecca, Medina, Taif, and Riyadh. In a linked list, Jeddah can only point to one city. In an array, Jeddah is just an index — you cannot encode "Jeddah is connected to Mecca and Taif" without a separate, complex structure.

Here is why graphs are the right tool for each domain:

1🚗

Road Networks

Cities are vertices, roads are edges. A city can have many roads (many neighbors) and each edge can store the distance. No linear structure can represent this naturally.

2👥

Social Networks

A person (vertex) can have hundreds of friends (edges). Friendship is mutual — the edge is undirected. "Mutual friends" is a graph intersection problem.

3🌐

Internet Routing

Routers (vertices) connect to several other routers (edges). Packets travel along graph paths. Dijkstra's algorithm finds the fastest route on a weighted graph.

Conclusion Some data by nature is about many-to-many relationships. Graphs are the only data structure designed precisely for this. When entities can be connected to multiple other entities in arbitrary ways — use a graph.

Types of Graphs

Graphs come in many varieties depending on the nature of their edges and connectivity. Understanding the types is essential for choosing the right algorithm.

Undirected vs Directed

In an undirected graph, edges have no direction — if A connects to B, then B also connects to A. In a directed graph (digraph), edges have a direction — an arrow from A to B does not imply B to A.

Directed

Weighted vs Unweighted

In an unweighted graph, all edges are equal. In a weighted graph, each edge carries a numeric value (cost, distance, capacity). Road maps use weighted graphs — the weight is the distance between cities.

Unweighted
Weighted

Cyclic vs Acyclic

A cyclic graph contains at least one cycle — a path that starts and ends at the same vertex. An acyclic graph has no cycles. Trees are acyclic. Road networks are typically cyclic.

Acyclic (no cycle)

Connected vs Disconnected

A connected graph has a path between every pair of vertices. A disconnected graph has at least two vertices with no path between them — it contains multiple connected components.

DAG — Directed Acyclic Graph

A DAG is a directed graph with no cycles. It is fundamental in task scheduling, compilers (dependency resolution), spreadsheets, and package managers. Topological sort — ordering tasks by dependencies — only works on DAGs.

DAG — No cycles, directions flow forward

Terminology

Before studying graph algorithms, you must be comfortable with the vocabulary. Every term below appears repeatedly in algorithm descriptions.

  • Vertex / Node
    A fundamental unit of a graph. Represents an entity: a city, a person, a computer, a task. The set of all vertices is denoted V.
    Example: In a flight map, each airport is a vertex.
  • Edge
    A connection between two vertices. Represents a relationship or link. The set of all edges is denoted E. An edge can be directed (one-way) or undirected (two-way).
    Example: A direct flight between two airports is an edge.
  • Degree
    The degree of a vertex is the number of edges connected to it. For directed graphs: in-degree = edges coming in, out-degree = edges going out.
    Example: If city D connects to B, C, and E — its degree is 3.
  • Path
    A path is a sequence of vertices where each consecutive pair is connected by an edge. A simple path visits no vertex more than once.
    Example: A → B → D → E is a path of length 3 (three edges).
  • Cycle
    A cycle is a path that starts and ends at the same vertex, passing through at least one other vertex. Detecting cycles is critical in deadlock detection and dependency resolution.
    Example: A → B → D → C → A is a cycle.
  • Adjacent
    Two vertices are adjacent (or neighbors) if there is an edge directly connecting them. In a directed graph, adjacency is directional: if edge (u,v) exists, v is adjacent to u but not necessarily u to v.
    Example: A and B are adjacent because edge (A,B) exists.
  • Connected Component
    A connected component is a maximal subset of vertices such that every vertex in the subset is reachable from every other vertex in the subset. A connected graph has exactly one component.
    Example: A disconnected graph with 6 vertices might have two components: {A,B,C} and {D,E,F}.
Quick Memory Aid Think of a graph as a city map. Vertices = intersections. Edges = roads. Degree = number of roads at an intersection. Path = a route from one place to another. Cycle = a road loop. Adjacent = directly connected by one road. Component = a region you can reach.

Adjacency Matrix

An adjacency matrix is one of the two main ways to represent a graph in memory. It uses a 2D array (matrix) of size V × V, where V is the number of vertices.

Rule: matrix[i][j] = 1 if there is an edge from vertex i to vertex j, otherwise matrix[i][j] = 0. For undirected graphs, the matrix is symmetric: matrix[i][j] = matrix[j][i].

Consider the same 5-node graph from Section 2: edges (A,B), (A,C), (B,D), (C,D), (D,E).

A B C D E
A 0 1 1 0 0
B 1 0 0 1 0
C 1 0 0 1 0
D 0 1 1 0 1
E 0 0 0 1 0

The blue 1s mark positions where an edge exists. Notice the matrix is symmetric along the diagonal — this is because the graph is undirected. The diagonal is all zeros because a vertex does not have an edge to itself.

Advantages Checking whether an edge (u, v) exists takes O(1) time — just look up matrix[u][v]. Adding or removing an edge is also O(1). Simple to implement.
Disadvantages Space usage is always O(V²) regardless of how many edges exist. For a graph with 1000 vertices, this means a 1,000,000-cell matrix — most of which may be zero for a sparse graph. Iterating over all neighbors of a vertex takes O(V) time, not O(degree).

Adjacency List

An adjacency list represents the graph by storing, for each vertex, a list of its neighbors. Instead of a full V×V matrix, we only store the edges that actually exist.

Structure: An array (or dictionary) of size V. Each entry is a list (linked list, dynamic array, etc.) containing the vertices adjacent to that vertex.

For the same graph — edges (A,B), (A,C), (B,D), (C,D), (D,E):

A --> [ B, C ] B --> [ A, D ] C --> [ A, D ] D --> [ B, C, E ] E --> [ D ]

Each key is a vertex and each value is the list of its direct neighbors. Vertex D has three neighbors (B, C, E) so its list has three entries. Vertex E has only one neighbor (D), so its list has one entry.

Advantages Space usage is O(V + E) — only actual edges are stored. Far more efficient than a matrix for sparse graphs. Iterating over all neighbors of a vertex takes O(degree) time, not O(V). This is what most graph algorithms (DFS, BFS, Dijkstra's) rely on.
Disadvantages Checking whether a specific edge (u, v) exists requires scanning u's adjacency list: O(degree(u)) time. For dense graphs (many edges), this can be slower than the matrix's O(1) lookup.
In Practice Most real-world graphs are sparse — a person on Facebook has hundreds of friends, not billions. A city has dozens of roads, not connections to every other city on Earth. For sparse graphs, the adjacency list almost always outperforms the adjacency matrix.

Adjacency Matrix vs Adjacency List

Choosing the right representation affects the performance of every graph algorithm you run. Here is a direct comparison:

Feature Adjacency Matrix Adjacency List
Space Complexity O(V²) — always allocates V×V cells O(V + E) — only stores existing edges
Check edge (u, v) O(1) — direct array lookup O(degree(u)) — must scan neighbor list
Add an edge O(1) — set two cells to 1 O(1) — append to adjacency list
Remove an edge O(1) — set two cells to 0 O(degree(u)) — find and remove from list
Iterate all neighbors of u O(V) — must scan entire row, even zeros O(degree(u)) — only visits real neighbors
Best suited for Dense graphs (E close to V²), or when frequent edge-existence checks are needed Sparse graphs (E much less than V²), which is most real-world graphs
Memory with V=1000 1,000,000 cells (even if only 2,000 edges exist) ~4,000 entries (2 × E for undirected graph with 2000 edges)
Rule of Thumb Use an adjacency list as your default choice. Switch to a matrix only when: (a) the graph is dense, or (b) you need O(1) edge existence checks and memory is not a concern. DFS, BFS, Dijkstra's, and most classic algorithms are designed around adjacency lists.

Depth-First Search (DFS)

Explores as deep as possible before backtracking. Uses a stack. Finds paths, detects cycles, and classifies edges.

Breadth-First Search (BFS)

Explores all neighbors before going deeper. Uses a queue. Finds shortest paths in unweighted graphs.
